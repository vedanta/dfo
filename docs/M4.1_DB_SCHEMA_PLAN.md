## Design Decisions

### Option A: Separate Tables ✅ RECOMMENDED

**Approach:** Create dedicated table for each analysis type

**Pros:**
- ✅ Clear separation of concerns
- ✅ Each analysis has appropriate fields (no NULL columns)
- ✅ Easy to query specific analysis results
- ✅ Type-safe (no mixed data types)
- ✅ Can optimize indexes per analysis
- ✅ Easy to drop/recreate individual analysis results

**Cons:**
- ⚠️ More tables to manage (7 total)
- ⚠️ Reporting needs to join multiple tables (acceptable)

### Option B: Unified Table ❌ NOT RECOMMENDED

**Approach:** Single `vm_analysis` table with `analysis_type` discriminator

**Pros:**
- ✅ Single table for all analyses
- ✅ Simpler reporting (one query)

**Cons:**
- ❌ Sparse table (many NULL fields)
- ❌ Less type safety
- ❌ More complex queries (filter by analysis_type)
- ❌ Harder to maintain as analyses grow
- ❌ Schema changes affect all analyses

**DECISION: Use Option A (Separate Tables)**

## Detailed Table Design

### Table 1: vm_low_cpu_analysis

**Purpose:** VMs with consistently low CPU utilization (candidates for downsizing)

**Note:** Uses underscores to match Python module name `low_cpu.py` (see DATABASE_CONVENTIONS.md)

```sql
CREATE TABLE IF NOT EXISTS vm_low_cpu_analysis (
    vm_id TEXT,                          -- FK to vm_inventory.vm_id
    cpu_avg DOUBLE,                      -- Average CPU utilization (e.g., 12.5 for 12.5%)
    days_under_threshold INTEGER,        -- Number of days below threshold
    current_sku TEXT,                    -- Current VM size (e.g., "Standard_D4s_v3")
    recommended_sku TEXT,                -- Recommended smaller size (e.g., "Standard_D2s_v3")
    current_monthly_cost DOUBLE,         -- Current monthly cost
    recommended_monthly_cost DOUBLE,     -- Recommended SKU monthly cost
    estimated_monthly_savings DOUBLE,    -- Savings from downsizing
    savings_percentage DOUBLE,           -- Percentage savings (e.g., 50.0 for 50%)
    severity TEXT,                       -- critical/high/medium/low
    analyzed_at TIMESTAMP                -- When analysis was run
);
```

**Field Explanations:**

| Field | Type | Purpose | Example |
|-------|------|---------|---------|
| `vm_id` | TEXT | Unique VM identifier | `/subscriptions/.../vm-web-01` |
| `cpu_avg` | DOUBLE | Average CPU over period | `12.5` (12.5%) |
| `days_under_threshold` | INTEGER | Days below 20% CPU | `14` |
| `current_sku` | TEXT | Current VM size | `Standard_D4s_v3` |
| `recommended_sku` | TEXT | Recommended size | `Standard_D2s_v3` |
| `current_monthly_cost` | DOUBLE | Current cost | `140.16` |
| `recommended_monthly_cost` | DOUBLE | New cost | `70.08` |
| `estimated_monthly_savings` | DOUBLE | Savings amount | `70.08` |
| `savings_percentage` | DOUBLE | Savings percent | `50.0` |
| `severity` | TEXT | Impact level | `medium` |
| `analyzed_at` | TIMESTAMP | Analysis time | `2025-01-24 10:30:00` |

**Why These Fields:**
- `cpu_avg` - Shows utilization for context
- `days_under_threshold` - Proves consistent low usage
- `current_sku` + `recommended_sku` - Clear before/after
- `current_monthly_cost` + `recommended_monthly_cost` - Transparency
- `estimated_monthly_savings` - Primary decision metric
- `savings_percentage` - Quick comparison metric
- `severity` - Prioritization

### Table 2: vm_stopped_vms_analysis

**Purpose:** VMs that have been stopped/deallocated for extended periods

**Note:** Uses underscores to match Python module name `stopped_vms.py` (see DATABASE_CONVENTIONS.md)

```sql
CREATE TABLE IF NOT EXISTS vm_stopped_vms_analysis (
    vm_id TEXT,                          -- FK to vm_inventory.vm_id
    power_state TEXT,                    -- 'stopped' or 'deallocated'
    days_stopped INTEGER,                -- Number of days in stopped state
    disk_cost_monthly DOUBLE,            -- Estimated monthly disk storage cost
    estimated_monthly_savings DOUBLE,    -- Savings from deletion
    severity TEXT,                       -- critical/high/medium/low
    recommended_action TEXT,             -- 'delete' or 'review'
    analyzed_at TIMESTAMP                -- When analysis was run
);
```

**Field Explanations:**

| Field | Type | Purpose | Example |
|-------|------|---------|---------|
| `vm_id` | TEXT | Unique VM identifier | `/subscriptions/.../vm-test-02` |
| `power_state` | TEXT | Current power state | `stopped` or `deallocated` |
| `days_stopped` | INTEGER | Days in stopped state | `45` |
| `disk_cost_monthly` | DOUBLE | Disk storage cost | `15.36` |
| `estimated_monthly_savings` | DOUBLE | Savings from deletion | `15.36` |
| `severity` | TEXT | Impact level | `low` |
| `recommended_action` | TEXT | What to do | `delete` |
| `analyzed_at` | TIMESTAMP | Analysis time | `2025-01-24 10:30:00` |

**Why These Fields:**
- `power_state` - Distinguish stopped vs deallocated
- `days_stopped` - Shows how long it's been unused
- `disk_cost_monthly` - Quantify the waste
- `estimated_monthly_savings` - Decision metric (equals disk cost)
- `severity` - Prioritization
- `recommended_action` - Clear guidance

**Note on Stopped vs Deallocated:**
- **Stopped:** VM is stopped but still allocated, incurs compute + disk costs
- **Deallocated:** VM is fully released, only incurs disk costs
- Both should be detected, deallocated has lower cost impact

## Complete Updated Schema

```sql
-- DuckDB schema for DevFinOps MVP

-- ============================================================================
-- INVENTORY TABLES
-- ============================================================================

-- VM discovery data and metrics
CREATE TABLE IF NOT EXISTS vm_inventory (
    vm_id TEXT,
    subscription_id TEXT,
    name TEXT,
    resource_group TEXT,
    location TEXT,
    size TEXT,
    power_state TEXT,
    os_type TEXT,
    priority TEXT,
    tags JSON,
    cpu_timeseries JSON,
    discovered_at TIMESTAMP
);

-- ============================================================================
-- ANALYSIS TABLES
-- ============================================================================

-- Analysis 1: Idle VM Detection
CREATE TABLE IF NOT EXISTS vm_idle_analysis (
    vm_id TEXT,
    cpu_avg DOUBLE,
    days_under_threshold INTEGER,
    estimated_monthly_savings DOUBLE,
    severity TEXT,
    recommended_action TEXT,
    equivalent_sku TEXT,
    analyzed_at TIMESTAMP
);

-- Analysis 2: Low-CPU (Right-Sizing) *** NEW ***
CREATE TABLE IF NOT EXISTS vm_low_cpu_analysis (
    vm_id TEXT,
    cpu_avg DOUBLE,
    days_under_threshold INTEGER,
    current_sku TEXT,
    recommended_sku TEXT,
    current_monthly_cost DOUBLE,
    recommended_monthly_cost DOUBLE,
    estimated_monthly_savings DOUBLE,
    savings_percentage DOUBLE,
    severity TEXT,
    analyzed_at TIMESTAMP
);

-- Analysis 3: Stopped VM Detection *** NEW ***
CREATE TABLE IF NOT EXISTS vm_stopped_vms_analysis (
    vm_id TEXT,
    power_state TEXT,
    days_stopped INTEGER,
    disk_cost_monthly DOUBLE,
    estimated_monthly_savings DOUBLE,
    severity TEXT,
    recommended_action TEXT,
    analyzed_at TIMESTAMP
);

-- ============================================================================
-- EXECUTION TABLES
-- ============================================================================

-- Action execution logs
CREATE TABLE IF NOT EXISTS vm_actions (
    vm_id TEXT,
    action TEXT,
    status TEXT,
    dry_run BOOLEAN,
    executed_at TIMESTAMP,
    notes TEXT
);

-- ============================================================================
-- REFERENCE/CACHE TABLES
-- ============================================================================

-- VM pricing cache
CREATE TABLE IF NOT EXISTS vm_pricing_cache (
    vm_size TEXT,
    region TEXT,
    os_type TEXT,
    hourly_price DOUBLE,
    currency TEXT,
    fetched_at TIMESTAMP,
    PRIMARY KEY (vm_size, region, os_type)
);

-- Legacy to modern SKU mappings
CREATE TABLE IF NOT EXISTS vm_equivalence (
    legacy_sku TEXT PRIMARY KEY,
    modern_sku TEXT NOT NULL,
    vcpu_legacy INTEGER,
    vcpu_modern INTEGER,
    memory_gb_legacy DOUBLE,
    memory_gb_modern DOUBLE,
    series_family TEXT,
    notes TEXT
);
```

## Query Examples

### Get All Low-CPU VMs
```sql
SELECT
    i.name,
    i.resource_group,
    i.size AS current_sku,
    l.recommended_sku,
    l.cpu_avg,
    l.estimated_monthly_savings,
    l.severity
FROM vm_low_cpu_analysis l
JOIN vm_inventory i ON l.vm_id = i.vm_id
ORDER BY l.estimated_monthly_savings DESC;
```

### Get All Stopped VMs
```sql
SELECT
    i.name,
    i.resource_group,
    s.power_state,
    s.days_stopped,
    s.disk_cost_monthly,
    s.severity
FROM vm_stopped_vms_analysis s
JOIN vm_inventory i ON s.vm_id = i.vm_id
ORDER BY s.days_stopped DESC;
```

### Get All Optimization Opportunities (Cross-Analysis)
```sql
-- Union all analyses for portfolio view
SELECT
    'idle-vms' AS analysis_type,
    vm_id,
    estimated_monthly_savings,
    severity
FROM vm_idle_analysis

UNION ALL

SELECT
    'low-cpu' AS analysis_type,
    vm_id,
    estimated_monthly_savings,
    severity
FROM vm_low_cpu_analysis

UNION ALL

SELECT
    'stopped-vms' AS analysis_type,
    vm_id,
    estimated_monthly_savings,
    severity
FROM vm_stopped_vms_analysis

ORDER BY estimated_monthly_savings DESC;
```
