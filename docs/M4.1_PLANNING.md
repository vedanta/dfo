## Database Schema Considerations

### Current Tables

```sql
-- vm_inventory (existing)
CREATE TABLE vm_inventory (
    vm_id TEXT,
    name TEXT,
    resource_group TEXT,
    location TEXT,
    size TEXT,
    power_state TEXT,           -- 'running', 'stopped', 'deallocated'
    os_type TEXT,
    priority TEXT,
    tags JSON,
    cpu_timeseries JSON,        -- Used by idle-vms and low-cpu
    discovered_at TIMESTAMP
);

-- vm_idle_analysis (existing - for idle-vms)
CREATE TABLE vm_idle_analysis (
    vm_id TEXT,
    cpu_avg DOUBLE,
    days_under_threshold INTEGER,
    estimated_monthly_savings DOUBLE,
    severity TEXT,
    recommended_action TEXT,
    equivalent_sku TEXT,
    analyzed_at TIMESTAMP
);
```

### Options for New Analyses

**Option A: Separate Tables (Recommended)**
```sql
-- New table for low-cpu analysis
CREATE TABLE vm_lowcpu_analysis (
    vm_id TEXT,
    cpu_avg DOUBLE,
    days_under_threshold INTEGER,
    current_sku TEXT,
    recommended_sku TEXT,
    estimated_monthly_savings DOUBLE,
    severity TEXT,
    analyzed_at TIMESTAMP
);

-- New table for stopped-vms analysis
CREATE TABLE vm_stopped_analysis (
    vm_id TEXT,
    power_state TEXT,
    days_stopped INTEGER,
    disk_cost_monthly DOUBLE,
    estimated_monthly_savings DOUBLE,
    severity TEXT,
    recommended_action TEXT,
    analyzed_at TIMESTAMP
);
```

**Pros:**
- ✅ Clear separation of concerns
- ✅ Each analysis has appropriate fields
- ✅ Easy to query specific analysis results
- ✅ No schema conflicts

**Cons:**
- ⚠️ More tables to manage
- ⚠️ Reporting needs to query multiple tables

**Option B: Unified Table**
```sql
-- Rename vm_idle_analysis to vm_analysis
CREATE TABLE vm_analysis (
    vm_id TEXT,
    analysis_type TEXT,         -- 'idle-vms', 'low-cpu', 'stopped-vms'
    cpu_avg DOUBLE,             -- NULL for stopped-vms
    days_under_threshold INTEGER,
    current_sku TEXT,           -- NULL for idle-vms
    recommended_sku TEXT,       -- NULL for stopped-vms
    power_state TEXT,           -- NULL for idle-vms, low-cpu
    days_stopped INTEGER,       -- NULL for idle-vms, low-cpu
    estimated_monthly_savings DOUBLE,
    severity TEXT,
    recommended_action TEXT,
    analyzed_at TIMESTAMP
);
```

**Pros:**
- ✅ Single table for all analyses
- ✅ Easier reporting (one query)
- ✅ Simpler database schema

**Cons:**
- ⚠️ Sparse table (many NULL fields)
- ⚠️ Less type safety
- ⚠️ More complex queries (need to filter by analysis_type)

**RECOMMENDATION: Option A (Separate Tables)**
- Cleaner, more maintainable
- Better for future expansion
- Report module can join multiple tables

## SKU Recommendation Strategy

### For Low-CPU Analysis

**Use existing compute_mapper.py:**
```python
from dfo.analyze.compute_mapper import find_smaller_sku

# Example: Standard_D4s_v3 (4 vCPU) → Standard_D2s_v3 (2 vCPU)
current_sku = "Standard_D4s_v3"
cpu_avg = 15.0  # 15% average

# Find next size down in same family
recommended_sku = find_smaller_sku(current_sku)
# Returns: "Standard_D2s_v3"
```

**Logic:**
1. Stay in same VM family (Dsv3 → Dsv3)
2. Downsize by 1-2 steps (4 vCPU → 2 vCPU)
3. If already smallest, recommend migration to cheaper family
4. Consider CPU utilization (don't recommend too small)

## Severity Classification

### Unified Severity Levels

```python
def _determine_severity(estimated_savings: float) -> str:
    """Determine severity based on estimated monthly savings.

    Consistent across all analyses:
    - critical: >$500/month
    - high: $200-$500/month
    - medium: $50-$200/month
    - low: <$50/month
    """
    if estimated_savings >= 500:
        return "critical"
    elif estimated_savings >= 200:
        return "high"
    elif estimated_savings >= 50:
        return "medium"
    else:
        return "low"
```

**Same thresholds for:**
- idle-vms
- low-cpu
- stopped-vms
